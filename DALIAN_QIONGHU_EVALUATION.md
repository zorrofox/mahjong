# 大连穷胡玩法评估报告

> 参考来源：[百度百科 · 大连穷胡](https://baike.baidu.com/item/%E5%A4%A7%E8%BF%9E%E7%A9%B7%E8%83%A1/5734953)
> 评估日期：2026-03-02
> 当前实现：港式香港麻将（144 张，16 种番型）

---

## 一、大连穷胡规则概述

大连穷胡是辽宁省大连地区特有的四人麻将玩法，以"宝牌冲宝"为核心特色，并对胡牌手型设有严格约束（"穷胡"之名即源于此）。

### 1.1 基础设置

| 项目 | 规则 |
|---|---|
| 牌组 | 136 张（条/饼/万各 36 张 + 风牌 16 张 + 字牌 12 张，**无花牌/季牌**） |
| 玩家数 | 4 人 |
| 发牌 | 庄家 14 张，闲家各 13 张 |
| 起庄 | 首局随机，庄家胡牌或荒庄则连庄，闲家胡则下家坐庄 |

### 1.2 胡牌强制条件（全部满足才可胡）

| 条件 | 说明 | 豁免情形 |
|---|---|---|
| 三色齐全 | 手牌中条/饼/万三门均须有牌 | 无豁免 |
| 不缺幺九 | 手牌中须含 1 或 9 | 手牌含风/字牌时豁免 |
| 至少一刻子 | 明刻或暗刻均可 | 以中/發/白做将牌时豁免 |
| 禁止手把一 | 不可将所有牌全部吃碰杠，仅剩单钓 | 无豁免 |
| 非门清胡 | 门清（无吃碰杠）视为"吃番"而非禁止，但会额外加番给输家 | — |

> **注**：中/發/白（三元牌）只能做将牌，不可组刻子；风牌不可吃，三元牌不可碰（仅做将）。

### 1.3 宝牌系统

宝牌是大连穷胡的核心机制，其他玩法均无类似设计：

**宝牌产生时机**：
- 首个宣听的玩家掷骰子，从牌墙后方数若干墩，翻出该处下方牌作为本局宝牌，只有自己知道
- 后续宣听的玩家可查看宝牌，但看完后不得换听
- 若三张相同牌（宝牌共3张）全部被打出，须重新掷骰换宝

**冲宝（+2 番）**：
- 宣听后查看宝牌时，若宝牌恰好是自己当前听的牌，可立即直接胡牌（无需等待摸牌/放炮）

**摸宝（+1 番）**：
- 宣听后若再摸到宝牌，宝牌可替代任意所需牌张（类似万能牌），此时以摸宝胡牌

### 1.4 番型与加番规则

所有番数以"滚滚番"公式结算：`2^(总番-1) × 基本分`

| 番型 | 番数 | 说明 |
|---|---|---|
| 平胡（基础） | 1 | 任意合法胡牌均计 |
| 自摸 | +1 | 自己摸牌胡 |
| 放炮（点炮者额外） | +1 | 点炮者输分时多加 1 番 |
| 夹胡 | +1 | 宣听时处于夹张（两面只差一中间张） |
| 庄家 | +1 | 胡牌者为庄家时 |
| 未开门 | +1 | 胡牌者全程未吃碰杠（加在输家身上） |
| 三家门清 | +1 | 其余三家均未吃碰杠时 |
| 摸宝 | +1 | 听牌后摸到宝牌胡 |
| 冲宝 | +2 | 听牌后看宝即可直接胡 |
| 杠上开花 | +2 | 杠后补摸胡牌 |
| 抢杠胡 | +2 | 搶他人加杠的牌胡 |

### 1.5 结算规则

**放炮（荣和）**：
- 点炮者单独多输 1 番；其余两位闲家**不付分**（与港式"放炮全包"完全不同）
- 胡牌者收到的分 = 放炮者应付的一份

**自摸**：
- 三家各按当前总番付分，计算方式与港式相同

**杠分**：
- 明杠：三家各付 1 × 底注
- 暗杠：三家各付 2 × 底注
- 仅胡牌者（局末赢家）可结算杠分

**荒庄**：
- 牌墙仅剩 7 墩（14 张）时若无人胡牌，该局结束，**庄家不换**，不结算分数

### 1.6 截胡规则

多人同时听同一张牌，该牌出现时，逆时针方向距出牌者最近的玩家享有胡牌权，其余玩家不得胡。

### 1.7 听牌后的特殊权利

- 听牌后仍可杠牌，条件是杠完后手牌仍保持听牌状态
- 宣听/夹听后，下次摸牌前可查看宝牌；未听牌者不得看宝

## 二、与当前港式实现的核心差异

### 2.1 规则层差异对比

| 维度 | 港式（当前实现） | 大连穷胡 | 差异等级 |
|---|---|---|---|
| 牌组 | 144 张（含 8 张花牌） | 136 张（无花牌） | 🟡 中 |
| 胡牌约束 | 任意合法手型（MIN_HAN=1） | 三色全 + 幺九 + 至少一刻子 + 禁手把一 | 🔴 高 |
| 宝牌系统 | 无 | 冲宝/摸宝（万能牌 + 即时胡） | 🔴 高（全新机制） |
| 门清规则 | 门清荣和 +1 番 | 门清视为"吃番"，对手多输 | 🔴 高（逻辑反转） |
| 放炮结算 | 放炮者全包（3 人份额之和） | 放炮者仅多输 1 番，其余不付 | 🔴 高 |
| 夹胡番型 | 无 | +1 番 | 🟡 中 |
| 未开门加番 | 无 | 胡牌时未开门者多付 1 番 | 🟡 中 |
| 三家门清加番 | 无 | 三家均无副露 +1 番 | 🟡 中 |
| 冲宝/摸宝 | 无 | 冲宝+2 / 摸宝+1（需宝牌追踪）| 🔴 高（全新机制） |
| 截胡规则 | 无（同优先级取最近者） | 逆时针最近者独享胡牌权 | 🟢 低（已有距离逻辑） |
| 荒庄终止 | 牌墙摸尽 | 剩余 7 墩即结束 | 🟡 中 |
| 听牌后可杠 | 不限制 | 杠后须仍保持听牌状态 | 🟡 中 |
| 花牌本命花 | +1/张 | **不存在** | 🟡 中（需屏蔽） |
| 杠分 | 固定 1 筹码 | 明杠 1×底注 / 暗杠 2×底注 | 🟡 中 |
| 番数上限 | 7 番封顶 (CHIP_CAP=64) | 无明确封顶（滚滚番） | 🟡 中 |
| 三元牌规则 | 中發白可组刻子 | 中發白**只能做将**，不可碰 | 🔴 高 |

### 2.2 最关键的三处根本性差异

#### ① 放炮结算逻辑（颠覆性）

港式：放炮者独付「三人自摸份额之和」（全包），胡牌者向放炮者单独收款。
大连：放炮者仅**比自摸多输 1 番**，其余两家不付。相当于：
- 胡牌者收到的分 = 放炮者的单份自摸分（不是全包）
- 此逻辑导致荣和与自摸的赢家收益差距极小，策略上倾向于等自摸

#### ② 宝牌系统（全新有状态机制）

宝牌需要：私有信息（只有听牌者知道宝牌）、万能牌替换逻辑、即时胡牌（冲宝，不经过声索窗口）、宝牌计数（三张用完重新产生）。这是当前架构完全不具备的机制。

#### ③ 三元牌只能做将

当前 `hand.py` 允许中/發/白组成刻子；大连规则中三元牌仅能做对子将牌，此限制会渗透到胡牌判断、吃碰杠验证、AI 决策等多个层面。

## 四、工作量估算

### 4.1 按模块分级

| 模块 | 改动量 | 难度 | 预估工时 |
|---|---|---|---|
| `tiles.py` | 小（删花牌） | 低 | 0.5 天 |
| `hand.py` | 大（重写胡牌校验 + 番型计算 + 摸宝替换） | 高 | 4 天 |
| `game_state.py` | 大（听牌状态机 + 宝牌系统 + 荒庄） | 高 | 5 天 |
| `websocket.py` | 中（结算重写 + 截胡 + 私有信息） | 中 | 3 天 |
| `ai_player.py` | 中（听牌策略 + 宝牌决策） | 中 | 2 天 |
| 前端 | 中（宝牌UI + 听牌状态 + 番型文案） | 中 | 3 天 |
| 测试 | 大（重写所有 hand/game_state 测试） | 中 | 4 天 |
| **合计** | | | **约 21-25 天** |

### 4.2 复杂度热点

以下三个机制的实现复杂度最高，是整个工期的主要风险点：

1. **宝牌系统**（私有状态 + 万能替换 + 冲宝即时胡）：需在 WebSocket 层维护每位玩家的私有视图，同时处理宝牌万能替换时的胡牌枚举（需穷举宝牌可替换的所有位置）。

2. **胡牌前置校验**（三色全 + 幺九 + 至少一刻子）：这些约束需要在胡牌判断、`get_available_actions`（声索窗口）、AI 听牌决策三处保持一致，任何不一致都会产生假阳性/阴性。参考港式麻将 Bug #8、#15、#23 的历史教训，这类一致性问题很容易引入隐蔽 bug。

3. **放炮结算语义反转**：大连的"放炮者多 1 番，其余不付"与港式的"放炮全包"逻辑方向相反，需要完全重写 `_pay()` 函数，且自摸/荣和的番数计算基准也不同（大连荣和番数 = 自摸番数，而非港式的独立计算）。

## 五、兼容性方案建议

### 方案 A：独立分支（推荐）

新建 `dalian` 分支，完整实现大连穷胡规则，与 `main`（港式）并行维护。

```
main   → 港式香港麻将（当前）
dalian → 大连穷胡（新实现）
```

**优点**：两套规则互不干扰，可独立部署、独立测试。
**缺点**：代码重复，bug 修复需同步两处。

### 方案 B：房间级规则配置

在房间创建时增加 `ruleset` 参数（`"hk"` / `"dalian"`），各核心模块通过规则集分发到不同实现。

```python
# 示例
class GameState:
    def __init__(self, ..., ruleset: str = "hk"):
        self.ruleset = ruleset
        self.validator = HKValidator() if ruleset == "hk" else DalianValidator()
```

**优点**：单一代码库，玩家可在大厅选择玩法。
**缺点**：架构复杂度显著上升，条件分支散落全局，增加维护难度；宝牌系统与港式花牌系统共用状态机时容易产生冲突。

### 方案 C：先出原型验证（低风险启动）

优先实现核心差异中的**非宝牌部分**（调整胡牌约束 + 结算公式），验证基础规则可玩性后，再单独迭代宝牌系统。

```
迭代 1（约 10 天）：
  - 移除花牌
  - 胡牌前置条件（三色/幺九/至少一刻）
  - 三元牌只做将
  - 结算公式调整（放炮 / 自摸 / 庄家 / 未开门）
  - 荒庄 7 墩终止

迭代 2（约 10 天）：
  - 宝牌系统（私有信息 + 摸宝 + 冲宝）
  - 听牌状态机
  - 夹胡检测
  - 三家门清加番
```

## 六、主要技术风险

| 风险 | 概率 | 影响 | 缓解措施 |
|---|---|---|---|
| 宝牌私有状态在多 WebSocket 连接中泄露 | 中 | 高 | 每个 `game_state` 消息按玩家 ID 过滤，宝牌字段仅发送给听牌者 |
| 摸宝万能替换引发胡牌判断假阳性 | 高 | 高 | 单独为宝牌枚举所有可替换位置，每种替换后独立运行 `is_winning_hand_given_melds` 验证 |
| 三色全约束在副露后遗漏部分牌张 | 中 | 高 | 三色检查需合并 `concealed_tiles + declared_melds`，与港式 Bug #25 类似 |
| 听牌后杠导致听牌状态失效未校验 | 中 | 中 | 杠后调用 `get_available_actions` 重新验证听牌状态 |
| 现有 501 条测试全部基于港式规则 | — | 中 | 新规则测试套件需独立建立，不可复用港式断言 |

## 七、总结

大连穷胡与港式麻将在**结算哲学**（放炮不全包）、**胡牌约束**（严格的三色/幺九限制）、**宝牌系统**（私有信息 + 万能替换 + 即时冲宝）三个维度存在根本性差异，不是在现有代码上"调参"可以完成的，需要对 `hand.py`、`game_state.py`、`websocket.py` 进行深度重写。

**建议采用方案 A（独立分支）+ 方案 C（分两期迭代）的组合**：

1. 新建 `dalian` 分支
2. 迭代 1 优先实现无宝牌核心规则，尽早可玩验证
3. 迭代 2 补充宝牌系统，单独测试和验收
4. 稳定后再评估是否合入主库或以独立服务部署
